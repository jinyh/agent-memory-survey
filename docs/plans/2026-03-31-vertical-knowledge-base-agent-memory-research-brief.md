# 垂直知识库型 Agent Memory：研究简报

> status: draft | owner: Research Lead | 2026-03-31

## question_id

`RQ-001`

## 研究问题

当前 Agent Memory 研究的默认场景，大多集中在三类问题上：

- 聊天偏好记忆
- 任务状态记忆
- retrieval-heavy QA 场景中的历史召回

这些场景当然重要，但它们不足以覆盖另一类同样稳定且有研究价值的场景：**垂直知识库型 Agent**。在这类系统中，memory 的核心不再是“记住用户偏好”或“从历史里找回一段文本”，而是维护**知识状态**：哪些领域知识单元应进入长期状态、它们来自哪里、处于整体知识结构中的什么位置、以及如何面向不同读者组织解释。

以 `NewPowerSystem` 为代表样本，这类项目同时具备：

- 结构化专题文档
- 来源与证据登记
- 章节与学习路径组织
- 面向不同读者的解释任务
- 后续可被 skill lib 消费的知识状态层需求

因此，现有关于 Agent Memory 的讨论虽然覆盖了 retrieval、formation、evolution 的部分问题，但对这种**教学/解释/溯源/桥接型 memory-in-use** 的覆盖仍不稳定。核心问题不是“能否找回内容”，而是：

> **能否把事实、来源、脉络与读者适配视图组织成面向特定对象的解释性行为。**

本研究简报提出：**垂直知识库型 Agent Memory 应被识别为 Agent Memory 研究中的一个稳定案例类型**，并进入 survey 与后续 evaluation 视野。

## scope

本研究简报的范围限定为：

- 定义“垂直知识库型 Agent Memory”这一案例类型
- 说明该类型为何未被现有 Agent Memory 讨论稳定覆盖
- 使用 `NewPowerSystem` 作为代表样本，验证该类型不是抽象假设，而是可观察的真实系统形态
- 说明该案例类型如何扩展 `RQ-001` 的适用场景
- 说明它将如何影响后续 survey 分类与 evaluation 任务来源

本简报不进入代码实现，不直接提供 benchmark 设计，也不替代目标项目侧的 implementation plan。

## non_goals

- 不提出通用 Agent Memory 的大一统理论；本文只主张“垂直知识库型 Agent Memory”应成为一个稳定子类
- 不直接提出正式 benchmark 或 evaluation protocol；这些工作应留给后续工件
- 不替代 `NewPowerSystem` 项目侧的 memory 设计稿或 implementation plan

## success_criteria

1. 给出“垂直知识库型 Agent Memory”的清晰定义，而不是只停留在直觉描述
2. 证明 `NewPowerSystem` 满足该定义，说明这不是假想场景
3. 清楚说明该案例类型如何扩展 `RQ-001` 的适用场景
4. 为后续 `evidence-map` 与 `survey-update-note` 提供稳定输入
5. 形成一个可被 survey 正文引用的案例类型判断，而不只是 ideas note

## 案例类型定义

本文将“垂直知识库型 Agent Memory”定义为：

> 一类以**结构化领域知识源**为基础、以**来源追溯**与**脉络组织**为约束、并以**面向不同读者或使用场景的解释任务**为主要使用方式的 Agent Memory 形态。

要被识别为该案例类型，至少同时满足四条必要条件：

### 条件 1：存在结构化知识源

系统的主要记忆对象，不是零散对话，而是已经结构化组织的领域知识材料，例如：

- 专题文档
- 章节层级
- 来源库
- 术语系统
- 主题目录

### 条件 2：存在来源追溯要求

系统回答不仅要求“内容相关”，还要求能够说明：

- 依据来自哪里
- 哪些是权威来源
- 哪些是案例归纳或次级材料

### 条件 3：存在脉络组织要求

系统不能只返回孤立知识片段，还必须说明该知识单元在更大结构中的位置，例如：

- 位于哪篇文章或哪一章
- 与哪些主题相邻或递进
- 在整体学习路径中承担什么角色

### 条件 4：存在读者适配解释任务

同一事实或概念需要依据不同读者背景、目标或使用情境进行解释重写，而不是只输出统一答案。

## 反例边界

以下系统虽然可能带有 memory 机制，但不应直接归入“垂直知识库型 Agent Memory”：

### 反例 1：普通聊天偏好 memory

这类 memory 主要记录：

- 用户语气偏好
- 个人信息
- 输出格式偏好

它不以结构化知识源为核心，也不强调来源与脉络。

### 反例 2：不带来源与脉络的普通 RAG QA

这类系统可以检索相关文本，但回答主要是“找到相关 chunk”，并不要求：

- 说明知识在整体中的位置
- 对不同读者做系统性解释
- 稳定处理来源权重

### 反例 3：纯任务状态 memory

这类 memory 主要记录：

- 任务进度
- 工具调用结果
- 临时约束
- 当前计划状态

它可能很重要，但其核心不是领域知识状态。

## 代表样本：NewPowerSystem

本文以 `NewPowerSystem` 作为代表样本，原因不是它代表所有垂直系统，而是它已经明确满足上述四条必要条件。

### 对条件 1 的验证：存在结构化知识源

从 `NewPowerSystem/README.md` 与 `CLAUDE.md` 可见，该项目具有：

- `docs/zh/` 下分专题组织的知识文档
- A-G 七大领域结构
- 源材料清单（source registry）
- 受众画像与标准章节模板

这说明它的主要知识对象是结构化领域材料，而非临时会话历史。

### 对条件 2 的验证：存在来源追溯要求

`CLAUDE.md` 明确维护了 source registry，并区分：

- 官方文档
- 政策解读
- 实战案例
- 技术验证
- 网络资源

同时定义了“来源与争议”章节、权威来源优先与更新触发规则。这表明系统回答需要可追溯来源，而不仅是相关文本匹配。

### 对条件 3 的验证：存在脉络组织要求

`README.md` 已提供：

- 章节目录
- 分批学习路径
- 面向不同角色的推荐阅读路径

`CLAUDE.md` 进一步规定：

- 每篇文章有标准章节结构
- 文档属于不同文章类型（完整版、概念版、实践版、策展版）
- 内容应能关联到整体主题覆盖范围

这说明记忆单元在使用时必须带着章节与学习脉络被理解。

### 对条件 4 的验证：存在读者适配解释任务

`README.md` 与 `CLAUDE.md` 都明确指出，该项目服务三类读者：

- 电力行业从业者
- AI 技术爱好者/技术背景读者
- 跨学科研究者

并要求：

- 用类比建立直觉
- 做跨学科桥接
- 对不同背景的读者组织不同解释路径

因此，它不是单一答案系统，而是天然带有“同一事实，多种解释视图”的需求。

## 与 RQ-001 的关系

本文不将“垂直知识库型 Agent Memory”视为新的独立主线，而是将其视为 **`RQ-001` 的扩展案例**。

`RQ-001` 当前关注的是：

> 当前主流 Agent Memory benchmark 是否覆盖 formation 和 evolution 阶段，若未覆盖，缺口在哪里。

这条主线在现阶段主要从 benchmark、retrieval 与 lifecycle 评测切入。但 `NewPowerSystem` 这样的样本提醒我们：

- Agent Memory 的适用场景不只有 benchmark 式 QA 或任务状态管理
- 在垂直知识库场景中，formation / evolution / retrieval / memory-in-use 需要被重新翻译

因此，这一案例类型对 `RQ-001` 的主要贡献是：

### 主贡献：扩展适用场景

它说明 `RQ-001` 的讨论对象不应只停留在 retrieval-heavy 或对话型场景，还应包括：

- 以知识状态为核心的 memory 系统
- 以解释与教学为主要行为输出的 agent
- 强依赖来源与结构脉络的领域知识场景

### 次级贡献：扩展 memory-in-use 的理解

在该场景中，memory-in-use 不再只是“回忆并利用历史”，而是：

> 将事实、脉络、来源和解释视图组织成面向特定读者的教学型回答。

这使 `RQ-001` 中的 lifecycle 讨论获得了一个新的现实外延。

## 初始证据与判断依据

### 直接材料

- `NewPowerSystem/README.md`
- `NewPowerSystem/CLAUDE.md`
- `docs/plans/2026-03-30-domain-specific-memory-case-newpowersystem.md`

### 当前判断

1. `NewPowerSystem` 不是简单的 RAG 项目，而是垂直知识库型系统
2. 其 memory 重点应是知识状态，而非聊天偏好或临时任务状态
3. 该案例类型值得被 Agent Memory 研究显式识别
4. 该案例类型目前在主流 benchmark 讨论中缺少稳定位置

### 已知证据缺口

- 目前还缺少第二、第三个同类型样本来证明这不是单点特例
- 还缺少论文或系统工作可用于把这一案例类型与现有 memory taxonomy 对齐
- 还未将该案例类型系统回写到 survey 章节结构中

## 对 survey 的预期影响

这份 brief 的首要影响不在 evaluation，而在 survey。

它首先推动的是：

> 将“垂直知识库型 Agent Memory”加入 Agent Memory 的案例类型版图。

具体来说，后续 survey 回写应考虑：

1. 在案例版图或系统类型讨论中，为该类型保留明确位置
2. 不再默认把 memory 只理解为偏好记忆、任务状态记忆或 retrieval 历史
3. 在 memory-in-use 相关表述中，纳入教学、解释、桥接与导学行为

这一步完成后，才更适合进一步讨论它对 evaluation 的影响。

## 最终结论

本文提出：

> **垂直知识库型 Agent Memory 应被识别为 Agent Memory 研究中的一个稳定案例类型。**

其核心特征不是聊天偏好，也不是单层检索，而是：

- 以结构化知识源为基础
- 以来源追溯为约束
- 以脉络组织为必要条件
- 以读者适配解释任务为主要使用方式

`NewPowerSystem` 说明，这类系统已经在真实项目中存在。它既扩展了 `RQ-001` 的适用场景，也为后续 survey 与 evaluation 提供了新的研究入口。
